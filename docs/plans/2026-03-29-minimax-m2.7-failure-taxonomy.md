## Taxonomy

### 1. Hallucination Boundary And Graceful Refusal

Representative failures:

- `hallucination_3`
- `hallucination_5`
- `hallucination_11`
- `hallucination_13`
- `hallucination_17`
- `hallucination_23`
- `hallucination_25`
- `hallucination_27`

Representative success:

- `hallucination_1`

Common signal:

- `r_user_end_conversation = 0`

Observed pattern:

- The agent often detects the missing capability too late, or keeps the conversation alive after it should have cleanly acknowledged the missing tool / parameter / response and stopped escalating.
- In several hallucination failures, the agent drifts into alternative help, speculative lookups, or additional actions instead of staying inside the capability boundary.

Likely broken chain:

- detect unavailable capability
- explain limitation clearly
- avoid speculative workaround
- stop or safely contain the conversation

Design implication:

- Purple agent needs a dedicated "capability boundary" mode, not just generic reasoning.
- When a required tool / field / response is missing, the model should switch from problem-solving mode into constrained acknowledgment mode with strict exit rules.

### 2. Composite Vehicle-Control Executor Is Unstable

Representative failures:

- `base_5`
- `base_51`
- `base_53`
- `base_63`
- `base_67`
- `base_93`
- `disambiguation_7`
- `disambiguation_21`
- `disambiguation_27`
- `disambiguation_33`
- `disambiguation_43`

Representative success:

- `base_3`
- `disambiguation_1`

Common signal:

- `r_actions_final = 0`
- `r_actions_intermediate = 0`

Observed pattern:

- The agent broadly understands the user intent, but multi-action vehicle tasks degrade during execution.
- These tasks usually require a small plan with state inspection, policy-conditioned follow-up actions, and correct sequencing.
- The agent is weakest when a request implies bundled state changes rather than a single direct tool call.

Likely broken chain:

- infer hidden sub-actions
- fetch prerequisite state
- construct correct multi-step plan
- preserve valid ordering and parameter values

Design implication:

- Purple agent needs a deterministic or semi-deterministic action composer for climate, windows, seat heating, and light-control bundles.
- This should sit below the LLM policy understanding layer, so the model decides intent while the executor handles operational details.

### 3. Navigation State Machine And Route Mutation Grounding

Representative failures:

- `base_45`
- `base_55`
- `base_81`
- `base_83`
- `base_89`
- `base_97`
- `base_99`
- `disambiguation_29`
- `disambiguation_41`
- `disambiguation_45`
- `disambiguation_47`
- `disambiguation_49`

Common signal:

- `r_actions_final = 0`
- `r_actions_intermediate = 0`
- `r_tool_execution = 0`

Observed pattern:

- The agent struggles with active-navigation editing semantics.
- Typical failures include:
  - using `set_new_navigation` when navigation is already active
  - constructing invalid route IDs or route transitions
  - missing required route metadata such as `at_kilometer`
  - mismatching destination / waypoint relationships

Likely broken chain:

- read current navigation state
- choose correct edit primitive
- bind route / waypoint IDs consistently
- validate route topology before tool call

Design implication:

- Purple agent needs a navigation-specific state machine.
- This should normalize route editing into a few explicit operation types:
  - replace final destination
  - add waypoint
  - replace waypoint
  - delete waypoint
  - reset navigation

### 4. Missing Prerequisite Read / Evidence-Gathering Discipline

Representative failures:

- `base_39`
- `base_43`
- `base_85`
- `disambiguation_21`
- `disambiguation_27`
- `disambiguation_41`
- `disambiguation_49`

Common signal:

- `r_tool_subset = 0`

Observed pattern:

- The agent sometimes acts before gathering the specific evidence the evaluator expects.
- Concrete missed tools already observed:
  - `get_distance_by_soc`
  - `get_exterior_lights_status`
  - `get_charging_specs_and_status`

Likely broken chain:

- identify what information must be known first
- run the required read tools
- only then synthesize answer or action plan

Design implication:

- Purple agent needs a "precondition read planner" that is separate from final action planning.
- This planner should reason over required unknowns, not just over the final user request.

### 5. Policy / Formatting / Confirmation Discipline

Representative failures:

- `base_37`
- `base_61`
- `base_77`
- `disambiguation_5`
- `disambiguation_23`
- `disambiguation_39`

Common signal:

- `r_policy = 0`

Observed pattern:

- The agent often gets the action roughly right but fails the policy layer through wording or process.
- Recurrent policy misses:
  - 12-hour time formatting instead of 24-hour time
  - saying "degrees" instead of explicitly saying Celsius
  - calling `send_email` without explicit confirmation workflow
  - choosing the fastest route correctly but failing to explicitly say that it proactively did so

Likely broken chain:

- generate content
- policy-format lint
- rewrite response before final send

Design implication:

- This is a strong case for a deterministic response linter / rewriter after model generation.
- A small formatting and confirmation policy layer should be cheaper and more reliable than asking the model to remember every wording rule in-context.

### 6. Parameter / Schema Grounding Is Brittle

Representative failures:

- `hallucination_19`
- `hallucination_33`
- `hallucination_37`
- `hallucination_43`
- `hallucination_45`
- `base_97`
- `disambiguation_47`
- `disambiguation_49`

Common signal:

- `r_tool_execution = 0`

Observed pattern:

- The agent emits tool calls that are conceptually related to the user goal, but structurally invalid.
- Concrete examples already observed:
  - invalid `contact_id`
  - invalid destination format such as `loc_madrid`
  - route / waypoint mismatch
  - missing `at_kilometer`
  - unknown start/destination type combinations

Likely broken chain:

- derive candidate arguments
- normalize to canonical schema
- validate against tool contract
- only then emit the call

Design implication:

- Purple agent needs a tool-argument normalization and validation layer.
- This should be explicit and schema-driven, not left entirely to the base model.

## What We Can Diagnose From The Stored Logs

For both successful and failed cases, the stored artifacts usually preserve:

- the full user / assistant / tool trajectory for the task
- assistant `tool_calls`
- tool result messages
- assistant `reasoning_content` in many turns
- evaluator reward metrics
- policy and tool execution error strings when present

That means we can usually identify which chain broke:

- clarification
- prerequisite read
- tool selection
- parameter grounding
- sequencing
- policy wording
- hallucination containment

## What We Cannot Fully Diagnose From These Logs

The stored task artifacts are not full raw provider logs. They do not reliably preserve:

- provider request IDs
- raw HTTP payloads / headers
- per-call latency breakdown
- hidden retries inside model wrappers
- token-by-token streaming traces

So the answer is:

- yes, the saved traces are complete enough for failure taxonomy and chain-level diagnosis
- no, they are not complete enough for transport-level or provider-level debugging
