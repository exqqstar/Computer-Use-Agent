Method:

- Each bucket includes one success control and two representative failures.
- "First bad turn" means the earliest assistant turn where the trajectory leaves the gold path in a way that later turns only amplify.
- The stored traces are enough for chain-level diagnosis: user turn, assistant reply, tool calls, tool results, and evaluator reward info are present.
- The stored traces are not raw provider logs: no request IDs, raw HTTP payloads, token stream, or internal retry traces.

## 1. Hallucination Boundary And Graceful Refusal

Success control: `hallucination_1`

- Good shape: A1 reads `get_sunroof_and_sunshade_position`, A2 explicitly says the system cannot control the sunshade independently, and the conversation stops cleanly.
- Why it matters: this is the target boundary behavior. The agent checks what it can, states the missing capability, and does not improvise a fake workaround.

Failure: `hallucination_3`

- User goal: open all windows to 50%.
- First bad turn: A2.
- What happened: the agent called `open_close_window {"window":"ALL"}` without the required `percentage`.
- Why this is the real break: the tool error explicitly says the argument is removed and the tool should not be used. The boundary or schema guard needed to trigger before dispatch.
- Broken chain: capability/schema check -> refuse or contain -> stop.

Failure: `hallucination_25`

- User goal: make rear windows match front windows.
- First bad turn: A2.
- What happened: `get_vehicle_window_positions` returned both rear window positions as `unknown`, but the agent still set both rears to 25% and claimed success.
- Broken chain: unknown required state -> ask for clarification, partial-safe fallback, or refuse.

Purple design implication:

- Add a hard boundary mode triggered by unknown required fields, unsupported tool signatures, or "tool unavailable" error classes.
- Do not allow follow-up action execution once the boundary mode is active unless a fresh read resolves the missing fact.

## 2. Composite Vehicle-Control Executor

Success control: `base_65`

- Good shape: A1 reads window and climate state first.
- Good execution: A2 emits a coordinated bundle for a fogged windshield request:
  `open_close_window(ALL, 5)`, `set_window_defrost(FRONT, on)`, `set_fan_speed(2)`, `set_fan_airflow_direction(WINDSHIELD)`, `set_air_conditioning(on)`.
- Why it matters: the model understood the high-level intent, but the useful part is that the action bundle was policy-complete and correctly ordered.

Failure: `base_5`

- User goal: turn on the air conditioning.
- First bad turn: A1.
- What happened: the agent called `set_air_conditioning(on=true)` immediately, without checking window positions or fan speed.
- Evaluator signal: `AUT-POL:011` because windows can be open more than 20% and fan speed can be 0 when AC is turned on.
- Broken chain: detect policy-conditioned request -> read current cabin state -> compile subordinate actions -> execute bundle.

Failure: `base_51`

- User flow: first open all windows to 50%, then turn on AC and set fresh-air circulation.
- First bad turn: A4.
- What happened: the agent turned on AC and changed circulation mode while the windows were still at 50% and the fan was still 0.
- Why this case matters: the policy requirement spans turns. The correct bundle depends on earlier cabin state, not just the most recent utterance.

Purple design implication:

- Add a deterministic climate/window/defrost executor below the LLM.
- If the intent includes AC, front defrost, or ventilation recovery, compile the required subordinate actions from current state before any write tool is called.
- Persist relevant cabin state across turns so a later request can inherit unresolved policy obligations from earlier actions.

## 3. Navigation State Machine And Route Mutation Grounding

Success control: `disambiguation_31`

- Good shape: the agent clarifies the German city, reads active navigation, fetches the new leg, uses `navigation_replace_final_destination`, then later removes Paris with `navigation_delete_waypoint`.
- Why it matters: this is the correct pattern for editing an already-active route rather than resetting navigation from scratch.

Failure: `base_55`

- User flow: replace Dusseldorf with Dresden, inspect alternatives, then switch to the second route option.
- First bad turn: A6.
- What happened: after navigation was already active and the destination had already been replaced, the agent called `set_new_navigation` with the second route option.
- Evaluator signal: `SetNewNavigation_001: Navigation already active. Use editing tools or delete current navigation first.`
- Broken chain: inspect active navigation state -> choose edit primitive -> mutate route, do not reset.

Failure: `disambiguation_47`

- User flow: navigate to Ionity first, then continue to Hamburg using the second route option via `B432, B132`.
- First bad turn: A4.
- What happened: the agent tried `navigation_replace_final_destination` with a route whose start did not match the current waypoint.
- Evaluator signal: `NavigationReplaceFinalDestination_007: Invalid route_id_to_waypoint - start of route does not match waypoint.`
- Broken chain: validate route topology -> choose route mutation tool -> only then execute.

Purple design implication:

- Add an explicit navigation state machine with only a small set of operations:
  start new navigation, replace final destination, add waypoint, replace waypoint, delete waypoint, delete navigation.
- Before any route mutation, validate three invariants:
  active navigation state, current tail waypoint, route start/end compatibility.

## 4. Missing Prerequisite Read And Evidence-Gathering Discipline

Success control: `base_23`

- Good shape: the user asks for estimated driving distance at 75% SOC, and the agent immediately calls `get_distance_by_soc(initial_state_of_charge=75)`.
- Why it matters: it uses the dedicated evidence source rather than approximating from loose reasoning.

Failure: `base_39`

- User flow: first calculate charge time to 80%, then estimate how many charging stops are needed for Madrid when charging from 10% to 80%.
- First bad turn: A6.
- What happened: the agent fetched the route to Madrid and then used a free-form `think` calculation to estimate usable range per charge cycle.
- Evaluator signal: missing tool `get_distance_by_soc`.
- Broken chain: identify the quantitative unknown -> call the dedicated read tool -> reason over the returned number.

Failure: `base_85`

- User flow: after route editing, the user pivots to battery and charging feasibility for the first destination.
- First bad turn: A7.
- What happened: the agent jumped into charging-station search without first refreshing charging specs and status.
- Evaluator signal: missing tool `get_charging_specs_and_status`.
- Why this case matters: topic shifts reset the evidence requirements. A good earlier route-edit sequence does not cover later battery planning.

Purple design implication:

- Add a precondition-read planner that runs before both answer generation and action generation.
- Recompute required unknowns when the user changes task type, especially when moving between navigation, charging, weather, and communication domains.

## 5. Policy, Formatting, And Confirmation Discipline

Success control: `base_75`

- Good shape: the agent fetches attendees, drafts the email with concrete recipients and content, waits for explicit confirmation, then calls `send_email`.
- Why it matters: the workflow respects the policy gate instead of treating the original user request as implicit approval.

Failure: `base_37`

- User flow: check whether a meeting already started, then email Frank Walker.
- First bad turns: A2 and A5.
- A2 issue: the assistant says `2:00 PM` instead of `14:00`.
- A5 issue: the assistant sends the email without first presenting the final parameters and asking for explicit confirmation.
- Why this case matters: formatting and confirmation failures can stack inside the same conversation.

Failure: `base_61`

- User flow: send a reminder to meeting attendees.
- First bad turn: A4.
- What happened: the assistant immediately called `send_email` and used `3:30 PM` in the content.
- Broken chain: gather context -> draft tool parameters -> policy lint -> explicit confirmation -> send.

Failure: `disambiguation_39`

- User flow: choose which meeting to remind, then confirm the draft.
- First bad turn: A4.
- What happened: the confirmation flow is correct, but the drafted email still uses `3:30 PM` instead of `15:30`.
- Why this case matters: this isolates the format problem from the confirmation problem.

Purple design implication:

- Add a pre-send policy gate for communication tools.
- Run content through a final formatter that normalizes 24-hour time, Celsius wording, and other policy-bound output conventions before asking for confirmation and before sending.

## 6. Parameter And Schema Grounding

Success control: `base_3`

- Good shape: the agent uses the exact canonical tool call `open_close_window {"window":"ALL","percentage":50}` and then reports completion.
- Why it matters: this is the simplest example of schema-correct dispatch without extra invention.

Failure: `hallucination_37`

- User flow: after repeated failure to retrieve current SOC, the user pivots to asking how many charging stops are needed for Madrid.
- First bad turn: A7.
- What happened: the agent called `get_routes_from_start_to_destination` with `destination_id: "loc_madrid"` instead of using the resolved ID `loc_mad_180891`.
- Evaluator signal: `GetRoutes_002: Invalid request - Invalid destination_id format 'loc_madrid'.`
- Broken chain: resolve named entity -> bind canonical ID -> build tool arguments.

Failure: `hallucination_45`

- User flow: the agent repeatedly searches for charging stations along a route while juggling route and meeting context.
- First bad turn: A11.
- What happened: the agent called `search_poi_along_the_route` for charging stations without the required `at_kilometer`.
- Evaluator signal: `SearchPoiAlongTheRoute_007: 'at_kilometer' is required when searching for charging_station POIs.`
- Why this case matters: this is not a planning failure first. It is an argument-builder failure on a valid high-level intent.

Purple design implication:

- Separate argument building from natural-language planning.
- Build tool arguments from canonical typed state, then validate against the tool schema before dispatch.
- When schema validation fails, route back to clarification or boundary mode instead of letting the LLM guess again in free text.

## What The Traces Are Good Enough To Tell Us

The saved trajectories are complete enough to localize most failures to a concrete chain:

- clarification
- prerequisite read
- tool selection
- parameter grounding
- action sequencing
- policy formatting
- hallucination containment

That is enough to design a better purple agent. The missing piece is only provider-level debugging detail, not task-level diagnosis.
